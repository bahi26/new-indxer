I can clarify things for you.</p><p>For one, the IV is not specifically related to AES at all.  AES is a keyed invertible transform from a 128 bit value to a 128 bit value; that's all it can do.  Now, if you just happen to have a 128 bit value that you want 'encrypted' into a 128 bit ciphertext, well, you can just use AES as is.</p><p>However, we typically want to other things than that; we may have plaintexts which aren't exactly 128 bits long; we might want nondeterministic encryption (so that if we encrypt the same value twice, it's not obvious that we did so); we might want integrity checking on our ciphertext as well.</p><p>To allow us to do those things, we use a <em>Mode of Operation</em>; that's a construction that uses AES (or some other block cipher) to perform these more generally useful operations.  It is these modes of operation which may (not all do) take an IV.</p><p>At the moment, the most popular modes of operation are CBC and GCM; they also have very different requirements to their IVs.</p><p>So, in that context, I'll answer your questions:</p><blockquote>  <p>I'm supposed to generate a different IV for every msg because using only a key I will get the same encryption if the message was encrypted twice</p></blockquote><p>That's not the complete story; yes, if you use the same IV, same message and same key (and same AAD for GCM), you'll get the same ciphertext, but that's not the full reason.</p><p>For CBC, if you use the same IV, there are two potential problems; one is that you'll leak how the two plaintexts are related.  For example, if the two plaintexts have the same first 16 bytes, and differ in the second 16 bytes, that'll be obvious to the attacker.  The other is that if the attacker can predict the IV (which he can if you use the same one repeatedly), and if he can introduce his own plaintext (which can happen in some scenarios), then he can use the encryption operation as an oracle (and thus deduce the value of low entropy plaintext blocks encrypted by the same key).</p><p>For GCM, if you use the same IV, there are two different potential problems (both of which are far more serious); for one, he can deduce the bit-wise xor of the two plaintexts (which is often enough for him to deduce those plaintexts).  Worse, it allows the attacker to deduce the internal authentication value, and thus allow him to modify ciphertexts (and thus the resulting plaintexts) without being detected.</p><blockquote>  <p>Now my question, is using the same IV for all of my messages is the same of not using IV at all? Or is it slightly better? </p></blockquote><p>Well, for neither mode, you don't have an option of 'not using an IV'; when you perform a CBC or GCM operation, an IV <em>must</em> be there (for GCM, you can have a zero length IV; that's not quite the same as 'not having an IV').</p><p>And, you didn't ask, but I'll state the general wisdom anyways:</p><ul><li><p>For CBC mode, the IV really ought to be unpredictable to the adversary.  Now, you can pick it randomly each time, or you can use (say) an AES encryption operation with the same key to generate it (which the attacker can't predict, as he doesn't have the key); both works.</p></li><li><p>For GCM mode, it doesn't matter in the least whether the IV is predictable to the adversary.  What's critical is that you use a different one each time.  If you can manage a counter (e.g. if you restart the program, you'll pick a fresh key), then using IV=0 for your first message, IV=1 for your second, etc, is the obvious approach.</p></li></ul> In fact, <strong>unless you are a cryptographer, you should forget that ‘AES’ itself exists as a thing</strong>: it is a <em>pseudorandom permutation family</em> $\operatorname{AES}_k\colon \{0,1\}^{128} \to \{0,1\}^{128}$, which is a technical jargon term that is practically meaningless to any application developer.</p><p>Instead, you should first identify what you are trying to do, such as send messages—arbitrary-length bit strings—from Alice to Bob who share a secret key.  You should also have in mind how many messages Alice and Bob will send (a handful? a few thousand? a few quadrillion?), and how long they might be (a kilobyte? a gigabyte? an exabyte?).  Then you should identify the adversary's capabilities, such as either eavesdropping on or possibly interfering with the channel from Alice to Bob.  Then you should decide what security properties you want:  Should the adversary be prevented from <em>reading</em> messages?  From <em>forging</em> messages?</p><p><strong>Once you have these <em>written down in a design document</em> for your application, <em>then</em> you can think about how crypto can help you.</strong></p><p>Usually what you want is to make sure that an adversary can't read <em>or</em> forge messages, unless you have a <em>very compelling reason</em> to argue that the adversary doesn't even have the opportunity to forge messages in the first place (example: disk encryption on your laptop—if customs takes your laptop out of sight at the border, you don't really want to use that laptop again) or sell analytics about your users' conversations to authoritarian fascist regimes.</p><p>To thwart an adversary who can eavesdrop on and meddle with the channel between Alice and Bob, you should <strong>pick an authenticated-encryption (with associated data) scheme, or AE(AD)</strong>.</p><p>An authenticated encryption scheme at the sender's end takes a key and a message (an arbitrary bit string) and some other parameters and returns a ciphertext, and at the receiver's end takes a key and a ciphertext and some other parameters and either screams <em>FORGERY!!!1</em> or returns a message.</p><p>Here's an example of an authenticated encryption scheme: AES-GCM.  Don't read AES-GCM as ‘the block cipher^Wpseudorandom permutation family AES, in the Galois/Counter mode of operation’—<strong>read AES-GCM as the following contract</strong>:</p><blockquote>  <p>This is a contract, between you, an application developer, and AES-GCM, a spell cast by high wizards of cryptography endorsed by the United States federal government and other lesser empires.</p>    <p><strong>Your obligations:</strong></p>    <ol>  <li>You must pick a 256-bit secret key $k$ uniformly at random and keep it secret.</li>  <li>For each message you send under the same key $k$, you must use a unique 96-bit <strong>nonce</strong> $n$ (sometimes also called initialization vector) which sender and receiver agree on for each message.  (You may transmit it alongside the message, or you might infer it from context, such as a message sequence number.  If Alice and Bob exchange messages in both directions, they must not choose the same nonce, so make sure, <em>e.g.,</em> Alice uses even nonces and Bob uses odd nonces.)</li>  <li>You must not send more than $2^{68}$ bytes of data total using the same key $k$.</li>  <li>If you pick $n$ randomly, you must not send more than $2^{32}$ messages using the same key $k$.  If you pick $n$ sequentially, you must not send more than $2^{48}$ messages using the same key $k$.</li>  </ol>    <p><strong>IN EXCHANGE, AES-GCM guarantees</strong>, against an adversary who can <em>adaptively influence</em> plaintexts sent by Alice and Bob and ciphertexts opened by Alice and Bob,</p>    <ol>  <li>The adversary cannot <em>distinguish the ciphertexts of two equal-length messages</em> even of their choice sent between Alice and Bob, and thus cannot <em>read messages</em> chosen by Alice or Bob.</li>  <li>The adversary cannot <em>forge messages</em> not chosen by Alice or Bob.</li>  </ol>    <p>If you violate any terms of this contract, the security properties provided by AES-GCM shall be rendered null and void.</p></blockquote><p>The security contract for AES-CBC is a little bit different, partly because <strong>AES-CBC does not provide authentication</strong>, which are words that should scare the ever-living daylights out of you.  AES-CBC is <em>only</em> an encryption scheme, not an authenticated-encryption scheme.</p><blockquote>  <p>This is a contract between you, an application developer, and AES-CBC, a meager spell cast by early wizards of cryptography who ill-understood the consequences or persistence of the magic they were weaving in a bygone era of primordial cryptosorcery before the modern foundations on whose shoulders we now stand were built.</p>    <p><strong>Your obligations:</strong></p>    <ol>  <li>You must pick a 256-bit secret key $k$ uniformly at random and keep it secret.</li>  <li>You must ensure your messages are always an integer multiple of 128 bits long.</li>  <li>For each message you send under the same key $k$, you must choose a 128-bit <strong>initialization vector</strong> that is (a) <strong>unique</strong> and (b) <strong>unpredictable in advance</strong>.  (You may transmit it alongside the message, or you might derive it from context, such as by a pseudorandom function of the message sequence number under another secret key shared by sender and receiver.)</li>  <li>You must not send more than $2^{68}$ bytes of data total using the same key $k$.</li>  <li>You must not send more than $2^{48}$ messages using the same key $k$.</li>  </ol>    <p><strong>IN EXCHANGE, AES-CBC guarantees</strong>, against an adversary who can <em>adaptively influence</em> plaintexts sent by Alice and Bob,</p>    <ol>  <li>The adversary cannot <em>distinguish the ciphertexts of two equal-length messages</em> even of their choice sent between Alice and Bob, and thus cannot <em>read messages</em> chosen by Alice or Bob.</li>  <li>That's all.  You don't get any more out of this contract, buddy.</li>  </ol>    <p>If you violate term (3)(a) of your obligations, the adversary can tell when two ciphertexts conceal the same messages.  If you violate any <em>other</em> terms of this contract, the security properties provided by AES-CBC shall be rendered null and void.</p></blockquote><p>There are other widely deployed authenticated encryption schemes out there, such as <a href="https://nacl.cr.yp.to/secretbox.html" rel="nofollow noreferrer">NaCl crypto_secretbox_xsalsa20poly1305</a> or libsodium's variants, which are all-around safer than AES-GCM but have fewer imperial endorsements.  There are also other spells built out of AES, sometimes colloquially called ‘modes of operation’ by cryptographers who value job security over the ability of anyone outside the high priesthood to handle cryptographic spells and user privacy and secrets and integrity, but this is not the forum for a catalog of all the spells—you need to <strong>write that design document first before we can say more about what crypto you should use</strong> if an authenticated encryption scheme is not what you need.</p><p>Finally, to answer the question you asked: It does not make sense to fail to provide an initialization vector for an operation that takes one.</p><p>But there are some APIs that are well-designed to maximize the confusion of dear readers like you by providing an endless array of hyperconfigurable options to pick a block or stream cipher and which cipher and key length and key and mode of operation and initialization vector and nonce and authentication tag and padding scheme and message encoding and message—all this to season your indecipherable acronym soup.</p><p>In these APIs, if you fail to provide some optional function parameter specifying an IV, or if you don't call the <code>set_iv</code> method or what-have-you, gnomes behind the scenes might silently choose an IV for you.  I recommend avoiding these APIs, or, if you must use them, treating them like nuclear waste requiring high degrees of protective gear and hand-holding by experts.</p>    </div>    <div class="grid mb0 AES has two inputs: a 128, 192 or 256 bit key and a 128 bit block of input. It generates a 128 bit block of output.</p><p>For each key each input block translates to a specific output block during encryption and the same mapping is reversed during decryption. This is called a keyed permutation as the key defines which input blocks are mapped to specific output blocks.</p><p>AES is a block cipher, but a block cipher cannot be used as a generic cipher. Encrypting the same block twice will result in the same ciphertext - leaking information to an adversary - and you can only encrypt 128 bit messages.</p><p>For this reason AES - as any block cipher - needs to be used <em>as a function</em> in a mode of operation. To this mode of operation is usually relatively simple compared to the block cipher; so we say that AES is used in a mode of operation.</p><p>An initialization vector (IV) or number-used-once (nonce) - I'll use just IV - is used in most modes of operation to generate a ciphertext indistinguishable from random. The specific requirements of the IV depend almost entirely on the mode of operation. The only factor of the block cipher that influences the IV value is the <em>block size</em>, which is fixed at 128 bits for AES.</p><p>An IV can have the following properties that depend on the mode of operation:</p><ul><li>the size of the IV may differ, e.g. CBC always requires a 16 byte IV of the full block size while GCM defaults to a 12 byte IV;</li><li>the IV may need to be indistinguishable from random to an adversary, or it may just need to be unique (for a specific key);</li><li>choosing an IV may influence how many messages or blocks can be encrypted securely by the mode of operation and block cipher;</li></ul><p>If the IV is of sufficient size then you may not need to store any information about the IV : you can just generate a <em>secure random</em> IV and keep it with the ciphertext. As the IV is large enough the chances of generating the same IV is small enough not to be a problem. In special cases, such as the 12 byte GCM IV, this may however limit the number of messages that should be encrypted with the same IV.</p><p>If the IV just needs to be unique then it may be possible to reuse an existing counter or identifier. For instance, disk encryption often use the specific sector as IV (or tweak, see notes). Messages in transport protocols often also contain unique sequence numbers. It is also possible to encrypt such information using a block cipher to generate an IV indistinguishable from random to an adversary. In that case it may not be needed to send the IV separately.</p><p>Usually the number of messages or blocks that may be encrypted is so high that these actual values are ignored. However, sometimes there are real limits to be taken into account. For instance using a 64 bit block cipher such as triple DES will pose you definite limits; DES can hardly be used in counter mode encryption. GCM will also pose higher but reachable limits to how much data can be encrypted with a specific key.</p><hr><p>Notes:</p><ul><li>The IV does not need to be kept secret. </li><li>Some block ciphers also accept a <em>tweak</em> and are called tweakable block ciphers, these are used in specialized constructs such as secure hashes, the Threefish block cipher in the Skein hash function is such a tweakable block cipher.</li><li>Information about the <em>size</em> and <em>timing</em> of the message may still be leaked; the mode of operation will not (and cannot, for indefinite length messages) protect against that.</li><li>Many times the mode of operation is left out as it is assumed that AES will provide all the security. Quite often this means that AES is run in CBC mode.</li><li>Sometimes the IV/nonce is not all that well defined, for instance CTR/SIC mode (counter mode) doesn't specify how the counter is generated from the required nonce - most implementations simply require you to generate the initial 128 bit counter value and call that the IV - which means that you can shoot yourself in the foot if the counter values are not far enough apart.</li><li>There are deterministic modes of encryption that do not require you to input an IV; one is called SIV mode or synthetic IV mode, here the IV is calculated by performing an operation over all of the bits of the message - even if a single bit differs the SIV mode will generate a unique IV, but the uniqueness must still be present <em>in the message itself</em>.</li><li>In AEAD authenticated modes of operation such as GCM the IV does <em>not</em> need to be protected separately (put in the authenticated data) as the mode of operation itself will take care of that; If a separate message authentication code is calculated over the ciphertext then the calculation <em>should</em> also include the IV.</li></ul>    </div>    <div
